# Spott 跨端核心旅程 UI 设计

日期：2026-07-16  
状态：已批准，进入实施计划  
适用范围：Web、iOS、API、OpenAPI、数据库、三语、无障碍、自动化与视觉验收

## 1. 目标与定位

本批交付 Spott 第一条可独立验收的双端核心旅程：

> 发现活动 → 查看活动详情 → 报名或加入候补 → 在行程中确认后续动作

它必须同时在 Web 与 iOS 可用，并以真实数据、完整中日英、发布级无障碍和真实渲染截图证明质量。目标不是重画现有页面，而是让首次用户能够在 90 秒内完成一次可信报名，并在每一步看清时间、地点、费用、语言、名额、主办方和风险。

本批是“完成 Luma + Meetup 能力并超越其体验”总目标的第一个纵向批次，不缩减总目标。后续依次覆盖：

1. 创建活动、编辑、周期/混合活动与 Host Studio。
2. 票种、优惠、支付、退款、会员和钱包票券。
3. 群组、讨论、活动聊天、私信、相册与关系沉淀。
4. 团队角色、营销、日历、洞察、品牌网络与公共 API。
5. AI 三语创建、可信翻译、个性化推荐、候补优化和安全工具。
6. Web Ops 对上述领域的完整运营与治理。

## 2. 设计依据

### 2.1 仓库事实

- Web 已有发现、活动详情和报名页面，但日期与余位只过滤当前页，地图缺失，部分 aria 和文案固定中文，视觉原语没有形成稳定组件层。
- iOS 已有发现、详情和报名，但地图坐标由数组下标生成，非发现 Tab 的活动跳转可能写入隐藏的 `discoveryPath`，三语动态字符串和无障碍覆盖不完整。
- 数据库已有 PostGIS `events.event_locations.point` 与 GiST 索引，但 API 没有输出真实坐标，也没有完整实现契约中已有的 `startsAfter`、`bounds` 查询。
- `packages/design-tokens` 已定义浅/深色语义色、4/8 间距、语义圆角、字体层级和 120/220/360ms 动效；Web 与 iOS 当前仍有各自漂移值。

### 2.2 竞品基线

本批不复制 Luma 或 Meetup 的受版权保护界面。竞品只作为能力与结果基线：

- Luma 的优势是清晰活动页、快速报名、日历/主办方品牌、票务、提醒和现场签到。
- Meetup 的优势是地区/兴趣发现、群组关系、周期活动、社区证明和持续参与。
- Spott 的领先点是日本本地语义、三语同一产品、精确地址保护、风险披露、候补与履约状态、跨端一致后续动作。

## 3. 实施路径选择

评估过三种路径：

1. **全站视觉换肤**：短期截图多，但会保留假坐标、错误筛选、断裂导航和死入口，不采用。
2. **先建完整设计系统再改页面**：一致性最好，但用户较晚看到完整结果，不采用为主路径。
3. **双端纵向旅程 + 随用随建原语**：每批交付真实闭环，同时沉淀可复用组件；采用。

每个任务必须产生一个可运行、可测试、可截图的增量。不能用静态 Mock 页面替代真实页面，也不能把未实现能力画成可点击按钮。

## 4. 产品信息架构

### 4.1 Web

桌面保留全局 Header，但减少竞争性入口：

- 左：Spott 品牌与当前地区。
- 中：发现、社群、我的行程、主办方。
- 右：语言、通知、账户、唯一强调的“创建活动”。

移动端保留五项 Dock：发现、社群、创建、行程、我的。Dock 必须尊重安全区，不遮挡页面底部主操作；当前路由需有可感知选中态。

核心旅程 URL 保持稳定且可分享：

- `/discover?...`：所有筛选、排序与地图范围反映在查询参数中。
- `/e/{slug}`：公开活动详情。
- `/register/{slug}`：登录/手机号验证后回到原报名上下文。
- `/me/events`：报名完成后的行程与状态中心。

浏览器返回必须恢复筛选、滚动和选中活动，不把用户送回空白首页。

### 4.2 iOS

保留五个原生 Tab：发现、行程、创建、社群、我的。每个 Tab 拥有独立 `NavigationPath`，任意来源点击活动都在当前可见栈打开详情；跨 Tab 深链先切换目标 Tab，再追加路由。

报名需要登录或手机号验证时，Gate 完成后恢复同一报名意图，不要求用户重新搜索活动。详情页返回后恢复列表/地图位置。

## 5. 视觉系统：Quiet Confidence

### 5.1 品牌语义

- 温暖画布和清晰纸面承载内容；浅色与深色都来自 `packages/design-tokens`。
- 暮紫只用于品牌、焦点和主操作；珊瑚用于少量提醒；薄荷用于真实成功或可用状态；琥珀用于候补/警告；危险红仅用于破坏性动作。
- 活动摄影是主要情绪来源。没有图片时使用安静的类别占位图，不生成假的活动摄影。
- 不堆叠玻璃、渐变、阴影和漂浮装饰。阴影只表达层级；普通卡片主要依赖背景、边界和空间。
- 中、日、英使用同一层级系统；不得为某一语言单独缩小字号以“塞进去”。

### 5.2 令牌与原语

Web CSS 和 `SpottTheme.swift` 必须映射同一套语义值：

- 间距：4、8、12、16、24、32、48、64。
- 圆角：control 12、card 18、cover 24、panel 28、pill 999。
- 动效：fast 120ms、standard 220ms、narrative 360ms；减少动态时降为 1ms。
- 类型：display、title、section、body、meta、caption，iOS 使用 Dynamic Type 对应语义样式，不固定到不可缩放字号。

首批沉淀以下跨页面视觉原语：

- Primary / Secondary / Quiet / Destructive Button。
- Filter Chip、Status Badge、Language Badge、Format Badge。
- Event Card、Event Facts、Organizer Trust Row。
- Remote Image（加载、失败、替代文本、焦点裁切）。
- Loading / Empty / Error / Offline / Permission / Success State。
- Sticky / Safe-area Action Bar。
- Field、Checkbox、Radio、Select、Inline Error、Form Summary。

## 6. 发现体验

### 6.1 首屏

移动端 360px 高度内必须至少出现一张真实活动卡的标题或封面，不能被超大 Hero、筛选面板或 Dock 完全占用。桌面首屏由紧凑品牌语句、搜索和首批活动组成，不制作独立营销首页来延迟发现。

搜索栏支持关键词；紧邻区域选择与列表/地图切换。常用筛选以可横向滚动 Chip 呈现，完整筛选进入 Sheet/Popover。

### 6.2 真实筛选

筛选必须在服务器、分页之前执行：

- 关键词 `q`。
- 地区 `region`。
- 分类 `category`。
- 开始区间 `startsAfter` / `startsBefore`，按活动 `displayTimeZone` 解释快捷日期。
- `availableOnly`，以容量和确认人数在查询时判断。
- 活动形态 `format`: `in_person | online | hybrid`。
- 主要/支持语言 `language`: `zh-Hans | ja | en`。
- 价格 `price`: `free | paid`。
- 地图范围 `bounds=west,south,east,north`。

游标排序固定为 `(starts_at, id)`。同一筛选组合的后续页不得出现重复、越过符合项或把符合项留在“看似空”的下一页。

### 6.3 活动卡

卡片只显示真实字段：

- 封面或类别占位。
- 本地化日期与时间。
- 标题。
- 地区/公开集合范围。
- 免费或日元价格。
- 活动形态与主要语言。
- 真实余位、候补、待审批或已报名状态。
- 主办方真实名称和可验证的信任属性。

没有真实参与者预览、共同好友或推荐模型时，不画假头像、不写“朋友参加”或“为你推荐”。个性化模块必须等真实解释字段到位后再启用。

### 6.4 地图与隐私

- iOS 使用 MapKit；Web 使用可替换的 MapLibre 适配层，样式 URL 由环境配置。
- 数据库 `point` 是内部精确点。发现结果永远只返回约 0.01° 网格精度的 `approximate` 坐标。
- 详情只有在现有地址权限允许时才返回 `exact` 坐标；否则仍为 `approximate`。
- 坐标响应包含 `precision: approximate | exact`，UI 明示“约在此区域”。
- 无坐标的活动仍可出现在列表，但不出现在地图；地图不得为其合成位置。
- Web 缺少地图样式配置时隐藏地图切换并保留路线外链，不显示破损空壳。
- 地图标点可打开活动预览并进入真实详情；列表和地图共享同一筛选状态。

## 7. 活动详情

### 7.1 首屏回答七个问题

首屏必须让用户无需滚动或只需轻微滚动即可回答：

1. 这是什么活动？
2. 什么时候？
3. 在哪里/线上哪里？
4. 免费还是多少钱？
5. 谁主办、可信依据是什么？
6. 还有多少名额，是否候补/审批？
7. 主要使用什么语言，支持什么语言？

Web 桌面采用内容主栏 + Sticky 报名卡；移动 Web 与 iOS 使用安全区底部状态栏。底栏不得遮挡最后一段内容。

### 7.2 内容顺序

1. 媒体与标题。
2. 时间、地点、形态、语言、费用、名额的事实区。
3. 主操作与当前报名状态。
4. 主办方和真实信任信息。
5. 活动说明、参与条件与报名问题预告。
6. 地址隐私、费用边界与取消规则。
7. 风险披露与安全入口。
8. 真实反馈摘要；无反馈时不展示空星级。

收藏、关注、分享、加入日历、路线和举报沿用真实动作，并有成功/失败反馈。主办方专属的未实现编辑动作不得用“入口已准备”占位。

### 7.3 CTA 状态

主操作根据真实状态展示一个明确动作：

- 立即报名。
- 申请参加。
- 加入候补。
- 继续完成登录。
- 继续手机验证。
- 查看待审批。
- 查看已报名行程。
- 接受候补名额。
- 已满且候补关闭。
- 报名截止、活动取消或活动结束。

颜色、图标和文本共同表达状态；不可只靠颜色。

## 8. 报名与完成态

### 8.1 原则

- 资料齐全的用户 90 秒内完成。
- 每一步只处理一类决策，输入失败后不丢失。
- 活动摘要始终可见，避免用户忘记正在报名哪场活动。
- 提交使用现有 quote 与幂等机制；按钮忙碌时禁止重复提交。

### 8.2 表单结构

1. **参与信息**：人数、真实报名问题、参与备注。
2. **规则确认**：参与条件、地址披露、候补/审批规则、取消边界。
3. **费用与提交**：线下费用、Spott 积分和最终结果。

短表单可在一个页面渐进分组；长表单使用可恢复步骤。Web 保持 URL 与浏览器返回可用，iOS 使用原生 Sheet/NavigationStack。

字段错误贴近字段，提交摘要把第一个错误带入视图并移动焦点/VoiceOver。服务端错误保留输入并提供重试。离线时不宣称报名成功。

### 8.3 完成态

成功不是短暂 Toast，而是完整确认页/Sheet，展示：

- 已确认、待审批或候补位置。
- 活动标题、日期、时间、公开地点、人数。
- 下一步：加入日历、分享、查看行程。
- 已确认用户可查看地址规则与签到准备。
- 候补用户看清通知方式和接受期限。

完成后 Web 与 iOS 的“行程”读取同一服务端状态。

## 9. 行程确认

行程页不是本批全量重做对象，但必须完成核心闭环：

- 新报名立即出现在正确分组：待审批、候补、即将开始或过去。
- 从行程点击活动在当前可见导航栈打开详情。
- 状态卡提供唯一下一步，不出现无效按钮。
- 取消、接受候补、签到准备等已有真实动作继续可用。

## 10. 数据与契约

### 10.1 事件位置

OpenAPI 新增：

```yaml
EventCoordinate:
  type: object
  required: [latitude, longitude, precision]
  properties:
    latitude: { type: number, format: double, minimum: -90, maximum: 90 }
    longitude: { type: number, format: double, minimum: -180, maximum: 180 }
    precision: { type: string, enum: [approximate, exact] }
```

`EventSummary.coordinate` 只允许 `approximate`；`EventDetail.coordinate` 按地址权限输出 `approximate` 或 `exact`。无点时为 `null`。

### 10.2 活动形态与语言

事件模型增加真实字段：

```text
format: in_person | online | hybrid
primaryLocale: zh-Hans | ja | en
supportedLocales: 非空、去重、必须包含 primaryLocale、最多三项
```

现有事件迁移默认 `in_person`、`ja`、`[ja]`，但 UI 不把默认值描述为组织者已确认；旧活动显示“语言待主办方确认”，直到被显式保存。为此增加 `localeConfirmedAt` 可空字段或等价真实标记。

`localeConfirmedAt` 为空的旧活动不匹配显式语言筛选；否则迁移默认值会被错误解释为主办方确认过的日语活动。未确认活动仍出现在未指定语言的普通发现结果中，并显示“语言待确认”。

在线会议地址和精确线下地址不进入 `EventSummary`。本批只显示真实形态；在线加入权限与会议集成在后续 Host Studio 批次完成，不放死按钮。

### 10.3 主办方信任

删除 API 中硬编码的“手机号已验证”。详情只返回从身份/履约数据真实计算的字段：

```text
phoneVerified: boolean
completedEventCount: integer >= 0
attendanceRateBand: unavailable | under_70 | 70_89 | 90_plus
```

没有足够样本时使用 `unavailable`，UI 不展示虚构百分比或等级。

### 10.4 查询响应

`EventPage` 保留 `items`、`nextCursor`、`hasMore`、`serverTime`、`queryExplanationId`。客户端把筛选状态编码进 URL/请求；不再对已分页结果做会改变集合成员的日期或余位二次过滤。

## 11. 三语

- 产品 UI locale 固定支持 `zh-Hans`、`ja`、`en`，允许跟随系统。
- Web 可见文案、aria、metadata、空/错/离线/权限/成功状态全部进入集中资源，不使用组件内三元翻译。
- iOS 静态键使用 `LocalizedStringKey`；动态计算值使用 `String(localized:)` 或显式本地化映射，不能把中文运行时 String 当键碰运气。
- 日期、时间、JST、日元、人数、距离和复数使用当前 locale 格式化。
- 用户生成的标题和描述保持原文，并标记活动主要/支持语言；本批不伪装机器翻译。
- 三套键集合必须相等；缺键、空值和静默回退阻断合并。
- 日语 UI 在发布前需要母语审校；自动测试只能证明键和布局，不能代替语言质量审校。

## 12. 无障碍与响应式

### Web

- 所有交互目标至少 44×44 CSS px。
- 完整键盘操作、可见焦点、正确 heading/landmark、表单 label、错误关联和状态 live region。
- Sheet/Dialog 锁定焦点、Escape 关闭、关闭后恢复触发点。
- 200% 缩放、360/390/768/1024/1440px 无横向溢出、Dock/Sticky CTA 不遮挡。
- 尊重 `prefers-reduced-motion`、`prefers-contrast` 和系统深浅色。

### iOS

- 使用语义字体和 Dynamic Type；最大辅助字号仍可完成报名。
- VoiceOver 顺序与视觉顺序一致，卡片不能把所有信息压成不可理解的一长句。
- 直接操作区域至少 44×44pt，支持 Switch Control 和 Reduce Motion。
- 深浅色、横竖屏适用处和小/大设备均无截断。
- 地图提供等价列表，不能要求盲人只能操作地图标点。

## 13. 性能

- Web LCP < 2.5s、INP < 200ms、CLS < 0.1（真实移动网络基线）。
- 首屏图片提供正确尺寸与占位；不预加载列表所有图片。
- 地图库只在用户切换地图或地图进入可见区域时加载，不进入默认首屏关键包。
- iOS 列表滚动目标 60fps；图片解码不在 SwiftUI `body` 主线程同步执行。
- 搜索输入 250–350ms 防抖；旧请求取消，晚到响应不能覆盖新筛选。

## 14. 错误与降级

- 加载、空、错误、离线、权限、成功使用统一状态原语。
- API 失败保留上一批可用内容并标记刷新失败；不把错误展示为空结果。
- 地图无权限时仍能搜索地区和浏览列表。
- 无地图提供方配置时 Web 不显示地图按钮。
- 无坐标活动只显示公开地区文字。
- 登录/手机验证取消后回到活动详情，不丢失报名意图。
- 活动状态在表单期间变化时，提交返回真实冲突并刷新摘要；不自动改成成功。

## 15. 测试与真实渲染验收

### API / Contract

- 日期、余位、价格、语言、形态和 bounds 在分页前过滤。
- 同时间戳游标稳定，无重复和跳项。
- 发现坐标始终 approximate；未授权详情不能得到 exact；授权用户可得到 exact。
- 无点活动不合成坐标。
- 主办方信任字段来自真实数据库，不含硬编码文案。
- OpenAPI lint、bundle 和生成客户端漂移检查通过。

### Web

- 组件测试覆盖 Event Card、筛选 URL、CTA 状态和表单错误焦点。
- Playwright 覆盖访客发现 → 详情 → 登录恢复 → 报名 → 行程。
- 覆盖默认、加载、空、错误、离线、已满、候补、待审批、成功和活动变化。
- 三语 × 浅/深 × 390/768/1440 关键截图；键盘和 reduced motion 单独验证。

### iOS

- 单元测试覆盖每 Tab 独立路由、跨 Tab 深链、筛选编码和 CTA 状态。
- XCUITest 覆盖发现 → 详情 → Gate 恢复 → 报名 → 行程。
- 中日英、深浅色、最大 Dynamic Type、VoiceOver 关键路径和 Reduce Motion 有证据。
- 小屏与大屏 Simulator 保存基线截图。

### 人工体验门

- 390px Web 与标准 iPhone 首屏能看到真实活动内容。
- 首次用户在资料齐全时 90 秒内完成报名。
