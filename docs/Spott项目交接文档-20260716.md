### 1.3 不能混淆的三个概念

1. **代码存在不等于功能验收完成**：大量流程有真实代码和单元测试，但没有连接真实短信、邮件、APNs、StoreKit 生产配置或跨设备 E2E。
2. **测试通过不等于发布门禁通过**：当前构建与多数测试通过，但根 `pnpm check` 会在 API Lint 阶段失败。
3. **并行分支完成不等于主线完成**：`feat/core-journey-ui` 的提交与未提交代码必须合并并重新验证后，才能计入 `main`。

### 3.2 已经建立的架构能力

- PostgreSQL 作为业务权威事实源，客户端不直接连接数据库。
- 版本化 REST/OpenAPI、幂等键、游标分页、事务 Outbox 与增量同步骨架。
- 账号、活动、报名、候补、签到、群组、积分、安全、通知、运营、媒体与分析等领域 Schema。
- iOS 与 Web 使用同一 API，并具备账号合并、媒体恢复、群组转让、海报恢复等跨设备接口。
- API 与 Ops 分离安全边界；运营端具有 MFA/权限/审计/双人审批语义。
- iOS、Web、Ops 当前都采用浅色显示；Web/Ops 明确设置 `color-scheme: light`，iOS 当前强制 `.preferredColorScheme(.light)`。

需要纠正的文档偏差：根 README 提到 `docs/acceptance`、AWS 东京区域 IaC、CI 中的集成/E2E/负载/安全证据，但当前仓库没有 `.github` 工作流、`docs/acceptance` 或生产 Terraform/CloudFormation 文件。这些不能按 README 文案算作已交付。

## 7. 当前发布阻断项

### P0-1 Web Session 安全未实现

主线 `apps/web/app/lib/client-api.ts` 仍把 access token 和 refresh token 一起序列化到 `localStorage`。仓库中的 Web Session Security 文档只是一份未跟踪设计稿，生产代码、迁移、Cookie 路由、CSRF、并发 rotation、撤销和集成测试都尚未实现。

完成标准：Refresh Token 进入 Secure/HttpOnly/SameSite Cookie；Access Token 只在内存或 BFF；严格 Origin/CSRF；稳定 Session ID；多标签刷新锁；撤销即时生效；旧 localStorage 会话安全迁移；完整集成与攻击用例测试。

### P0-2 主线发现数据不可信

- iOS `EventMapView` 仍按事件数组索引生成东京中心附近的假经纬度。
- Web 主线只把关键词、地区、分类发送到 API，日期和余位在已加载的 24 条数据上二次过滤，因此分页结果和数量可能错误。
- 主线客户端仍存在展示默认值/CTA 推导与服务端事实漂移风险。

并行分支已提交发现契约和分页前筛选的主要修复，但 iOS 真实地图、Web 完整发现 UI、路线恢复与全旅程仍在开发，合并前不能视为已解决。

### P0-3 发布门禁是假绿

- 根 `pnpm check` 当前失败。
- API Lint 有 474 个错误。
- 根 Typecheck 没有真正覆盖 Web，Web 独立 TypeScript 检查失败。
- 没有 `.github` CI 工作流和合并保护证据。
- Design Tokens 没有测试；Web/显示测试大量是源码/SSR 静态断言，不等于真实浏览器 E2E。

### P0-4 跨端同步没有完成产品级验收

后端具有 `sync/pull`、`sync/push`、Outbox 和冲突结构，iOS 有 SyncEngine，但主线 Web 未消费同步接口。更严重的是，主线 `PersistenceStore.apply` 当前没有应用传入的 changes，却会推进同步 cursor；这可能造成“服务端变更未落地、本地却认为已经同步”的数据丢失。冷启动恢复 Session 时也没有完整重建用户同步 scope。该问题应当在所有视觉优化之前按 P0 修复。

以下 P0 场景也没有一套可重复的真实 E2E 证据：

- iOS 草稿在 Web 继续编辑并处理冲突。
- Web 报名后 iOS 立即看到同一状态。
- 最后一席并发只能成功一人，其余进入正确候补路径。
- 签到、积分、通知和运营处置跨端收敛且不重复。
- 运营下架后公开页、缓存、分享和精确地址同步收敛。

### P0-5 视觉与体验尚未获得最终验收

最新明确的用户要求是：

- Web 与 Ops 不要黑夜模式，必须清爽、精致、图标语义统一。
- iOS 使用原生 iOS 26 悬浮玻璃感；底部只能有一个导航层，不能重叠。
- 顶部按钮、搜索框和筛选项必须有可见但克制的玻璃边界。
- Logo 必须简洁高级，不能像临时 Demo。
- 双端功能可用性至少对齐 Luma/Meetup，并在日本本地化、信任安全和跨语言体验上更好。

当前代码已有一轮改造，但用户此前连续表示不满意，且没有签字确认的最终截图矩阵、Figma/视觉规范、可访问性与真机证据。因此只能标记为“持续优化/待产品验收”。

### P0-6 生产运行基础缺失

- 没有生产 IaC、CI/CD、分环境发布、灰度/回滚和可验证部署记录。
- 没有生产域名/DNS/TLS、监控、告警、SLO 看板、集中日志与 Trace 证据。
- 没有备份、恢复、RPO/RTO、灾难演练和数据保留执行证据。
- 当前本地仅 Web 3000 正在监听；Ops 3001 与 API 4100 在快照时未运行，不能将浏览器旧页面视为服务健康。

### P0-7 后端与运营安全/异步闭环不完整

- Ops Cookie 虽为 HttpOnly/Secure/SameSite，但 Refresh 有效期是 30 天，与“运营端无长期 Refresh Token”的安全目标冲突。
- Ops 登录只有邮箱 OTP；`mfa_enrolled_at` 只是资格标记，不是本次 Session 真正完成了 TOTP/Passkey 第二因子。高风险重新认证主要由 UI 禁用按钮表达，服务端没有对审批、付费积分调整和导出下载统一强制。
- 部分 Cookie 路由缺完整 CSRF/Origin 收口；数据库 RLS 对表所有者角色可能被绕过，`data_scopes` 也没有覆盖所有敏感运营域。
- 风险引擎没有把活动 `risk_flags` 可靠写入风险表；所有活动统一进入人工审核，没有自动通过/抽样/禁止分流，也没有完整的活动生命周期 Worker。
- 审核制报名 Hold 到期后可能只过期 Hold，不同步释放报名的 `pending_count`，存在名额永久占用风险。
- 媒体上传 Intent 不是生产 S3 预签名；普通媒体处理成功后没有完整推进 `moderation_state=approved`，活动提交也没有强制 ready/approved 封面。
- Ops 导出目前主要是审批和元数据状态，没有真实生成、上传、下载验证和到期删除 Worker。
- 非 console 的 OTP Provider 没有真实短信/邮件发送适配器；生产环境即使生成 Challenge，也可能无法把验证码送达用户。

## 8. 与“全面超过 Luma 与 Meetup”目标的明确差距

当前产品文档 V1 和用户后续目标之间存在范围冲突。开发文档明确不做 Android、开放式私信、无限群聊，也不由平台经手活动费；而“完整拥有 Luma 与 Meetup 的功能”通常会要求票务、会员、周期活动、消息与更强组织者工具。下一阶段必须由产品负责人决定是坚持 V1 边界，还是正式扩展产品文档，不能把冲突隐藏在“已完成”里。
