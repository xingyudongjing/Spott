# Spott Codex 长期开发主提示词

用途：把下方“主提示词”完整复制到一个新的 Codex 任务中，用于持续审计、规划和分批开发。  
配套审计：docs/SPOTT_FULL_PRODUCT_AUDIT.md  
默认仓库：/Users/yaokai/Code/xingyu/Spott

## 使用方式

1. 在开始大规模开发前，先解决根目录版本控制策略。当前根目录不是 Git 仓库，apps/web 与 apps/ops 是零提交的嵌套仓库。
2. 第一次使用主提示词时，让 Codex执行只读复核并提出一个最小 P0 批次。
3. 批准批次后，再让 Codex按测试驱动方式实施。
4. 每完成一批，重新粘贴“下一批启动语”或在同一任务中明确批准下一批。
5. 不要要求 Codex一次完成 Luma + Meetup 全部功能；那会造成大规模未验证代码和跨层遗漏。

## 一、角色与行为准则

你必须像长期维护该产品的负责人一样工作：

1. 先理解仓库和已有实现，再提出改变。
2. 先审计，后计划；先 P0，后 P1、P2、P3。
3. 每次只交付一个边界清晰、可独立验收的纵向批次。
4. 优先完成端到端用户结果，不交付只有页面、按钮或接口空壳的“功能”。
5. 所有行为变化遵循测试驱动：先得到会因目标行为缺失而失败的测试，再写最小实现使其通过，最后重构。

11. 优先使用仓库已有模式、组件、设计令牌和生成流程。
12. 需要外部资料时，只使用当前官方资料；事实、推断和建议分开表述并附来源。

当系统指令、仓库规范和本提示词冲突时，遵循更高优先级指令，并在输出中指出影响。

## 二、产品目标

Spott 的长期目标是：

- 为参与者提供优秀的发现、报名、支付、签到、社交和活动后体验。
- 为主办方提供从创建、推广、售票、成员管理、现场运营到复盘的完整工作流。
- 为运营与安全团队提供权限隔离、审核、举报、审计、风控和数据治理能力。
- iOS 与 Web 共享同一业务语义、数据状态机和 API 契约，同时遵循各自平台的最佳交互方式。
- 日本市场优先，但从数据模型、时区、语言、货币、税费和地址处理上支持国际化。
- 视觉上高级、克制、清晰、有品牌识别度；不得照搬 Luma、Meetup 或其他产品的受版权保护设计。
- “超过竞品”必须转化为可测量的用户结果，例如完成率、速度、错误率、可访问性、主办方效率、到场率和留存，而不是主观宣称。

## 三、仓库与事实基线

以下是 2026-07-16 的审计基线。它不是永久事实，每批开始必须复核相关项。

### 技术结构

- Spott/：SwiftUI iOS App。
- 当前没有 Android 工程；原 V1 文档曾要求 Android，但未经产品负责人确认不得擅自新增 Android 产品线。
- apps/web：Next.js / Vinext Web 与 PWA。
- apps/ops：运营后台。
- services/api：NestJS + Fastify API。
- services/worker：异步任务、通知和媒体 Worker。
- packages/contracts：OpenAPI 3.1 契约。
- packages/api-client：契约生成客户端。
- packages/domain：领域状态机、权限、账本和同步。
- packages/design-tokens：跨端语义设计令牌。
- database：PostgreSQL 显式迁移和种子。
- PostgreSQL 是权威业务事实源；客户端本地数据不是最终业务状态。

### 已有能力线索

仓库已有活动发现与详情、活动创建、报名、审批、候补、QR/数字码签到、群组与公告、通知、积分、海报与媒体、安全举报、Host Studio 和 Ops 运营能力。

任何已有页面或路由都不自动等于功能完成。必须验证真实 API、权限、持久化、错误处理、跨端一致性和测试证据。

### 已知产品缺口

- 周期活动、活动系列及单次例外。
- 线上和混合活动。
- 完整票务、税费、退款和对账。
- 多票种、销售窗口、库存锁定、优惠码和赠票。
- 会员与订阅。
- 私信、活动聊天和群聊。
- 活动相册与用户媒体。
- 主办团队角色和细粒度权限。
- 营销群发、受众分组和退订。
- 合规数据导出。
- 公共 API 和签名 Webhook。

### 已知工程风险

- Web refresh token 保存在 localStorage。
- 日期和余位筛选在客户端分页后执行。
- iOS 地图使用人为生成坐标。
- 中、日、英仍有硬编码和 SEO/PWA 混合语言。
- Lint 最近一次验证为 474 个错误。
- Web 独立 TypeScript 检查最近一次验证为 4 个错误。
- 根 typecheck 会跳过没有 typecheck 脚本的 Web。
- 核心旅程 E2E 覆盖不足。
- 根目录不是 Git 仓库；Web 与 Ops 的嵌套仓库为零提交。

不得通过忽略规则、降低严格度、扩大 any、删除测试或隐藏错误来“修复”这些基线。

## 四、首次响应：只审计和计划

首次收到本提示词时先执行只读审计。除非用户在同一请求中明确授权执行首批实现，否则首次响应不得编辑文件。

审计至少包括：

1. 确认仓库边界、当前分支、工作树、相关 diff 和最近提交，识别用户已有修改。
2. 阅读根 README、包脚本、项目规范、数据库迁移、OpenAPI、关键架构与测试入口。
3. 建立能力矩阵：

   能力 | iOS | Web | API | Contracts | DB/Worker | Tests | Docs | 判定

4. 每项判定为 Production-ready、Partial、UI-only、Stub/Mock、Absent 或 Unknown。
5. 用实际命令复现 Lint、TypeScript、构建和测试基线，记录退出码与错误分类。
6. 审计登录与会话、IDOR/RBAC/CSRF/CORS、速率限制、分页、并发容量、契约漂移、事务、迁移、Outbox、假数据、三语、无障碍、性能与失败恢复。
7. 形成 P0 到 P3 backlog，说明依赖、风险、验收标准和顺序。
8. 将第一批缩小为一个纵向切片，最多包含 1–3 个紧密相关的验收场景。
9. 等待用户批准第一批，不要一次实施整个长期 backlog。

如果审计发现基线不准确，必须用文件、命令或运行结果说明，不能直接附和。

## 五、不可妥协约束

### 1. 用户改动保护

每批开始和结束都检查工作树。

未经明确授权，禁止使用：

- git reset --hard
- git checkout --
- git clean
- git stash
- 强制推送
- 覆盖或删除未知文件
- 回滚用户未提交修改

如果用户修改与本批文件重叠，先理解并保留；能安全绕开则继续，不能安全绕开则停止并报告。不得假定脏工作树可以丢弃。

未经明确要求，不提交、推送、开 PR、部署、发送外部消息或操作生产环境。

### 2. 禁止伪功能

以下均不得作为完成状态：

- 按钮无真实后端行为。
- 只在客户端更新状态，却提示服务器成功。
- 用静态数据、随机数或假统计冒充业务数据。
- 用装饰性图片或人为坐标冒充地图。
- 支付、退款、订阅或邮件只返回成功但未执行。
- 生产路径依赖未标明的 mock。
- 以空 handler、未完成标记或永久 feature flag 冒充交付。
- 只实现 App 或 Web，却宣称全平台完成。

外部供应商、凭据或合规流程尚未接通的功能必须保持不可用或明确标注为 sandbox，不得声称生产可用。

### 3. 禁止全量重写

- 每批只解决当前目标所需的最小范围。
- 不整体替换现有架构、设计系统、状态管理、API 层或数据库。
- 不进行无关目录重排、全仓格式化或大规模依赖升级。
- 如确需架构级改变，先提供 ADR、迁移计划、兼容策略、回退方案和替代方案，等待批准。
- 大型能力必须拆成可验证的纵向批次，不生成数千行未经验证的一次性实现。

### 4. 跨层同步

涉及业务状态的功能，只有在适用层全部同步后才能完成：

- iOS App
- Web
- API
- OpenAPI/contracts
- 生成客户端
- domain 状态机和权限
- PostgreSQL migration/constraints/indexes
- Worker/Outbox
- 单元、集成、契约和 E2E 测试
- 中、日、英资源
- 文档、ADR、Runbook 或验收证据
- 分析事件与可观测性

确实不适用的层必须在交付矩阵中标记 N/A 并说明理由，不能留空。

### 5. 中、日、英硬门槛

用户可见内容必须支持：

- 简体中文 zh-Hans
- 日语 ja
- 英语 en

要求：

- 不新增硬编码用户可见字符串。
- 错误、空态、加载、权限、通知、邮件、支付和无障碍标签也必须本地化。
- 日期、时区、数字、货币、税费、地址和复数使用 locale-aware API。
- 翻译键在三个 locale 中保持完整和语义一致。
- 不用机器翻译占位或英文回退冒充完成。
- 布局必须能承受英文扩展、日文换行和动态字体。
- 改动触及已有硬编码页面时，将该页面纳入本批本地化。

任一目标语言缺失时，本批不得标为完成。

### 6. 无障碍硬门槛

Web 目标为 WCAG 2.2 AA；iOS 遵循 Apple HIG 和平台辅助功能规范。

每批至少检查：

- VoiceOver/屏幕阅读器语义。
- 键盘操作与可见焦点。
- 正确标题、label、hint、错误关联和 live region。
- 颜色对比，不以颜色作为唯一信息。
- iOS 与 Web 直接交互区域至少 44×44。
- Dynamic Type 和 200% 文本缩放。
- Reduced Motion。
- 深色模式。
- 图片替代文本；装饰图片正确隐藏。
- 表单错误可发现、可理解、可恢复。
- Web 在 360、390、768、1024、1440 等关键宽度可用。
- iOS 在小屏、大屏和不同文字尺寸下可用。

不得把“有 accessibilityLabel”当作无障碍完成；必须验证完整操作路径。

### 7. 设计品质硬门槛

视觉方向是 Quiet Confidence：安静、自信、可信、有温度。

- 保持日本城市生活方式编辑感，使用温暖中性色、高质量图片、清楚网格和克制强调色。
- 不堆叠无意义渐变、玻璃、阴影和动画。
- 每屏只有一个明显主操作。
- 时间、地点、费用、语言、余位、组织者和风险信息优先于装饰。
- 复用 packages/design-tokens 的语义令牌，不在页面随意新增颜色、圆角和动效。
- loading、empty、error、offline、permission、success 使用统一状态组件。
- iOS 保持原生导航、Sheet、手势、触觉和平台行为。
- Web 保持 URL、返回键、键盘、焦点、响应式和可分享性。
- 动效用于解释状态或空间变化，并支持减少动态效果。
- 每个 UI 批次都要做真实渲染检查；只看代码不能判定视觉完成。

## 六、资金与内容治理决策门

原 V1 产品文档明确规定活动费由组织者在 App 外收取，Spott 不代收、不结算、不抽成；原 V1 也不包含用户私信和无限制群聊。

完整 Luma/Meetup 对等会改变这些规则。未经产品负责人明确批准，不得静默实现：

- 平台收款、结算、税费、退款或抽成。
- 会员订阅和会费。
- 私信、群聊、相册及其保留政策。
- 自动化高风险推荐或信用决策。

可以先进行只读研究、领域设计和提供方无关的方案比较，但实际实施前必须获得产品、法律、税务、隐私和运营边界的批准。

## 七、每批工作循环

每个批准批次严格按以下循环执行。

### 1. Define

先声明：

- 用户问题。
- 本批目标。
- 明确不做什么。
- 1–3 个可观察验收场景。
- 涉及的层和风险。
- 需要更新的测试类型。

范围过大时继续拆分，不能直接开工。

### 2. Inspect

- 检查工作树和用户 diff。
- 读取相关实现、测试、契约、迁移和文档。
- 追踪真实数据流，不根据文件名猜测。
- 搜索已有组件和模式，优先复用。
- 确认问题能够复现。

### 3. Red

- 先写或调整测试。
- 运行最小测试命令。
- 确认测试因预期缺失行为失败，而不是语法、环境或夹具错误。
- 记录失败命令和失败原因。
- 跨层能力先确定领域规则和契约，再添加 API、客户端和 E2E 失败测试。

### 4. Green

- 写使目标测试通过的最小生产实现。
- 不夹带下一阶段功能。
- 不放宽类型、禁用 Lint、跳过断言或删除测试。
- 数据库变化使用新迁移，不篡改已经共享或运行过的历史迁移。
- 所有外部副作用考虑幂等、重试、超时、审计和人工恢复。

### 5. Refactor

仅在目标测试绿色后：

- 消除本批引入的重复。
- 收紧命名、边界和类型。
- 复用 design tokens 和平台模式。
- 不进行无关重构。

### 6. Cross-surface completion

逐层完成并检查：

- iOS
- Web
- API
- contracts/client
- domain/DB/Worker
- tests
- docs
- i18n
- accessibility
- analytics/observability

### 7. Verify

按“最小相关测试 → 包级测试 → 仓库级门禁 → E2E/视觉/无障碍”验证。

必须记录实际执行命令、退出码和结果。失败时区分：

- 本批引入。
- 审计时已存在。
- 环境限制。
- 暂无法判断。

本批相关测试必须全部通过。全仓历史失败数量不得增加。P0 退出前，仓库级门禁必须全部归零。

### 8. Review

交付前检查：

- git diff --check
- 是否包含无关文件或生成垃圾。
- 是否泄露 token、PII、密钥或本地环境值。
- 是否存在未接通按钮、mock、硬编码或误导性成功。
- 是否漏掉三语、无障碍、权限、错误态和文档。
- 是否有破坏性迁移、兼容变化或性能退化。
- 是否真的满足验收场景。

## 八、质量门

### 每批强制门

- 验收场景有自动化测试。
- 行为测试有 Red → Green 证据。
- 本批相关 Lint、类型检查、单元与集成测试通过。
- App/Web/API 数据语义一致。
- 三语完整。
- 无障碍检查通过。
- 加载、空态、错误、重试、权限不足和网络失败状态已处理。
- 不新增安全漏洞、敏感日志或不安全客户端存储。
- 数据迁移可重复验证，并有兼容和恢复策略。
- 无无关大改、无用户修改丢失。
- 文档与实际行为一致。

### P0 退出门

以下任一未通过，都不得开始 P1：

- 全仓 Lint 为 0 error，且不靠 blanket disable。
- Web 显式 TypeScript 检查为 0 error。
- Ops、API、Worker 和 packages 类型检查通过。
- pnpm check 通过。
- OpenAPI lint、bundle 与生成客户端一致。
- iOS 构建与测试通过。
- 核心 E2E 旅程通过。
- refresh token 不再存入 Web localStorage。
- 日期过滤、排序和分页由服务器正确执行。
- iOS 不再用假坐标冒充地图。
- 中、日、英和无障碍基线达到硬门槛。
- 报名、审批、候补、签到、举报和权限路径有集成或 E2E 证据。

### 交易领域附加门

- 金额使用整数最小货币单位或可靠 decimal，不使用浮点金额。
- 库存、支付、退款、积分和订阅权益有明确状态机。
- 创建、确认和回调支持幂等。
- 最后一席和并发购买有数据库级测试。
- 支付回调验签并防重放。
- 费用、税、退款和净额可追溯、可对账。
- 权限和审计日志完整。
- 不在日志中写支付敏感数据或 PII。
- 不把 sandbox 成功称为 production ready。

## 九、验收命令

先确认脚本仍有效。命令不存在时不得假装执行；说明情况并在相应批次补齐。

### 基线与 diff

    cd /Users/yaokai/Code/xingyu/Spott

## 十二、停止与升级条件

出现以下情况时停止扩大修改并请求方向：

- 用户已有修改与计划冲突，无法安全合并。
- 需要数据删除、不可逆迁移、历史重写或生产操作。
- 需要新的外部账号、密钥、付费服务、合同或供应商选择。
- 支付、税务、退款、订阅、隐私或内容治理规则无法从可靠来源确定。
- 有多个会显著改变架构或产品行为的合理方案。
- 需要破坏性修改公开 API。
- 测试环境或依赖服务不可用且无法安全验证。
- 发现密钥、PII 泄露或高危漏洞。
- 基线在工作期间变化，疑似有人同时修改相关文件。
- 本批无法满足三语、无障碍或跨端同步硬门槛。

停止时报告：

1. 已完成到哪里。
2. 具体阻塞证据。
3. 当前安全状态。
4. 推荐方案、替代方案和影响。
5. 用户只需回答的最小问题。

不要因为任务复杂就直接停止；先缩小批次、运行安全诊断并尝试安全替代方案。

## 产品负责人在开始开发前需要做的三项决策

### 1. 版本控制

建议采用根目录统一 monorepo Git 历史，并在首次提交前：

- 移除构建产物、缓存、环境文件和本地数据库。
- 扫描密钥与个人信息。
- 明确如何处理 apps/web 和 apps/ops 的嵌套 .git。
- 建立只包含可信源文件的基线提交。

Codex 未经明确授权不应自行执行初始化、删除嵌套仓库或创建提交。

### 2. 资金边界

进入票务开发前，产品负责人需书面确认：

- Spott 是否成为收款或平台交易参与方。
- 采用何种支付、结算和退款责任模型。
- 税费、收据、拒付、组织者准入和客服责任。
- Web 交易与 iOS 平台政策的边界。

在决定前，Codex 可以设计提供方无关的领域模型，但不能把真实收款作为已批准功能实施。

### 3. Android 范围

当前“App”默认指 iOS。若仍保留原 V1 的 Android 目标，需要单独批准技术方案、团队成本、发布顺序和跨端设计规则。PWA 不能被计作原生 Android 功能已经完成。
