# Spott 全面产品、设计与工程审计

审计日期：2026-07-16  
审计范围：iOS、Web/PWA、API、Worker、运营后台、OpenAPI、数据库迁移、设计令牌、三语资源、测试与本地运行  
产品目标：App 与 Web 均覆盖 Luma 和 Meetup 的核心能力，并在日本本地活动、跨语言体验、信任安全和视觉品质上形成明显优势

## 1. 执行结论

Spott 已经不是原型。它具备完整度较高的日本线下活动 V1 骨架：账号与手机号验证、活动发现和发布、报名审批、候补、签到、群组、通知、积分、海报、安全举报、组织者工作台、运营后台、OpenAPI 契约、异步 Worker 和跨端同步均已有真实代码。

但当前版本尚未达到以下三个目标：

1. 尚未达到 Luma 与 Meetup 的完整功能并集。
2. 尚未实现 App 与 Web 的严格能力对等。
3. 尚未达到可以用“比两者更高级、更精致、体验更好”对外宣传的质量门槛。

当前最准确的产品定位是：功能较深、架构方向正确、具有差异化雏形的日本本地活动平台 V1。下一阶段不应继续无序增加页面，而应先清除发布阻断项，再按活动运营、商业交易、社区关系、差异化体验四条产品线逐步完成。

### 2.1 “App”范围说明

当前仓库中的原生 App 是 iOS SwiftUI，没有 Android 工程。原 V1 非功能要求曾写明 iOS、Android 与移动 Web/PWA，但本轮用户要求只称“App 和 Web”。

本文默认近期 App 指现有 iOS App。若 Android 承诺仍然有效，则 Android 当前成熟度为 L0，应单独制定 Kotlin/Compose 或其他经批准方案，不能将 PWA 计作原生 Android 已完成，也不能由 Codex 在没有平台决策时擅自新建 Android 产品线。

## 4. P0：进入下一阶段开发前必须处理

### P0-1 iOS 地图展示了不真实的位置

证据：Spott/Features/Discovery/DiscoveryView.swift 的 EventMapView 根据列表序号，在东京中心坐标上人为偏移经纬度。

影响：

- 用户会被引导到错误区域。
- 这是数据真实性问题，不只是视觉问题。
- 任何基于距离、附近或地图浏览的体验都不可验收。

完成标准：

- EventSummary 和事件详情具备经过验证的经纬度与公开精度级别。
- 后端搜索支持地图边界或中心点半径。
- 未授权查看精确地点时，仅返回安全的近似坐标。
- 没有坐标的活动使用明确的列表降级，不生成假坐标。
- iOS 与 Web 使用同一地理字段和隐私规则。

### P0-2 Web 将 access token 和 refresh token 保存到 localStorage

证据：apps/web/app/lib/client-api.ts 的 WebSession 同时包含 accessToken 与 refreshToken，saveSession 将完整会话序列化到 localStorage。

影响：

- 页面发生 XSS 时，长期 refresh token 可被脚本读取并外传。
- 账户、组织者权限和运营行为都可能被接管。

完成标准：

- refresh token 改为 Secure、HttpOnly、SameSite Cookie。
- access token 只保留在内存或由同源 BFF 代理，不向第三方脚本暴露。
- 所有状态改变请求具备 CSRF 防护策略。
- 刷新、退出、撤销、设备会话和过期行为有集成测试。
- 迁移期间旧 localStorage 会话被安全清除，不把 token 写入日志或埋点。

### P0-3 Web 时间和余位筛选只作用于已加载的 24 条数据

证据：apps/web/app/components/DiscoverExperience.tsx 请求只发送关键词、地区和分类；日期与 availableOnly 在客户端对返回的当前页过滤。

影响：

- 符合日期条件的活动可能位于下一页，却被错误显示为空。
- 结果数量、分页和“加载更多”语义不可信。
- App 与 Web 的发现结果可能不一致。

完成标准：

- 日期范围、余位、线上/线下、价格、语言、距离等可搜索条件均由 API 执行。
- 游标基于完整排序和过滤条件生成。
- 同一查询在 iOS、Web 和直接 API 调用中返回一致集合。
- 属性测试覆盖跨时区、周末、满员、候补和分页边界。

### P0-4 发布门禁会产生“假绿”

本次验证结果：

- iOS Simulator 完整构建成功。
- pnpm build：9/9 任务成功。
- pnpm test：现有测试任务全部成功。
- pnpm typecheck 表面成功，但 apps/web 没有 typecheck 脚本，因此没有被根任务真正覆盖。
- Web 单独执行 tsc 失败，共 4 个错误：PWA icon purpose 类型、Cloudflare Workers 模块、Fetcher 和 D1Database 类型。
- pnpm lint 失败：API 共 474 个错误，主要为不安全 any、成员访问和测试替身类型。

完成标准：

- apps/web 和 apps/ops 都显式提供 typecheck。
- 根 pnpm check 覆盖每一个 workspace，不允许无脚本跳过。
- Lint、类型、单元、契约、构建、E2E 任一失败时 CI 必须失败。
- CI 输出能明确列出实际执行过的 package 和测试数量。
- 合并保护要求所有门禁成功。

### P0-5 当前仓库缺少可靠版本控制边界

证据：

- /Users/yaokai/Code/xingyu/Spott 根目录不是 Git 仓库。
- apps/web 与 apps/ops 各自存在 .git，但均为 main 分支零提交，所有文件未跟踪。
- iOS、API、数据库和共享 package 没有可见的统一历史。

影响：

- 无法可靠审查、回滚、建立 CI、追踪数据库迁移或保护用户改动。
- 多人或多代理开发极易覆盖彼此工作。

完成标准：

- 产品负责人确认单仓或多仓策略。
- 若保持当前 monorepo 结构，根目录建立统一 Git 历史，移除意外嵌套仓库。
- 第一个基线提交前扫描密钥、构建产物、DerivedData、环境文件与用户数据。
- 所有后续功能以小提交和可审查分支执行。

## 5. P1：上线质量与跨端一致性

### P1-1 三语不是完整闭环

现状：

- iOS 的 en、ja、zh-Hans Localizable.strings 各有 820 个键，键数量一致，这是良好基础。
- AppModel 的同步、缓存、成功和网络错误仍直接构造中文 String。
- SpottAPIClient 仍有中文上传和 HTTP 错误。
- AppLanguage 的“跟随系统”为中文动态字符串。
- Web 中央 messages.ts 只有约 80 个核心键，更多页面使用 locale 条件表达式或直接字面量。
- Web 页面 title、description、Open Graph locale 和图片 alt 固定为中文。
- Web manifest 内容为英文，但 lang 固定为 zh-Hans。

完成标准：

- 所有系统文案、错误、通知、邮件、SEO、PWA、无障碍标签均有中、日、英。
- CI 扫描 Swift 和 TypeScript 新增用户可见硬编码。
- 三语键集合完全一致；不允许静默回退后上线。
- 日期、时间、数字、货币、复数和地址按 locale 格式化。
- 用户生成内容保留原文，并允许显示机器翻译标记与原文切换。
- 日语文案由母语审校，不使用直译式 UI 语言。

### P1-2 Web 移动端尚未达到高级体验

浏览器证据：

- 桌面版视觉干净、留白和品牌色已有辨识度。
- 390×844 首屏 hero 文案占据约五行，首批活动被推到折叠线以下。
- 固定底部 dock 与筛选区域存在视觉遮挡风险。
- 多个直接交互控件高度约 32–39px，没有达到统一 44px 触控目标。
- 搜索、日语和英语切换可用，控制台无明显运行错误。

完成标准：

- 360px 宽度下无横向滚动、无底栏遮挡、主要内容首屏可见。
- 所有直接触控目标至少 44×44 CSS px，间距避免误触。
- 键盘导航、焦点可见、Escape、对话框焦点陷阱和屏幕阅读器标签完整。
- 200% 文本缩放不截断关键操作。
- 支持 prefers-reduced-motion 和高对比度。
- 每个核心页面在 360、390、768、1024、1440 宽度有截图回归。

### P1-3 iOS 缺少发布级无障碍与性能证据

现状：

- UI 测试仅覆盖五个底部导航入口、地图切换和未登录创建门禁。
- 核心报名、创建、审批、候补、签到、群组和设置没有 XCUITest。
- 应用强制 preferredColorScheme(.light)，现有暗色设计令牌未进入真实应用。
- 多处 UIImage(data:) 出现在 SwiftUI 视图路径，存在主线程解码和列表掉帧风险。
- 大量 AsyncImage 使用方式未形成统一缓存、降采样和占位策略。

完成标准：

- 支持 Dynamic Type 至最大辅助字号，关键操作不丢失。
- VoiceOver 顺序、标签、值、提示和自定义操作完整。
- 支持浅色、深色、提高对比度和减少动态效果。
- 图片在后台解码并按显示尺寸降采样。
- 列表滚动目标保持 60fps；性能回归由 Instruments/ETTrace 证据支持。
- XCUITest 覆盖核心参与者与组织者闭环。

### P1-4 本地持久化初始化失败会直接崩溃

证据：Spott/Core/Persistence/PersistenceStore.swift 的默认 SwiftData 初始化失败使用 fatalError。

完成标准：

- 区分可迁移、可重建缓存和不可恢复数据。
- 缓存损坏时安全重建；本地草稿先导出或迁移。
- 用户看到可恢复错误，而不是启动崩溃。
- 迁移、磁盘不足和损坏场景有测试。

### P1-5 开发环境与生产元数据边界不清

本地 Web 开发服务器出现过 fetch failed；生产构建服务可运行。metadataBase 完全依赖 NEXT_PUBLIC_SITE_URL，本地配置曾指向运营端端口，导致公开页面资源和分享元数据地址错误。

完成标准：

- dev、test、preview、production 使用经过 schema 验证的环境配置。
- Web、Ops、API URL 不允许端口或域名互相指错。
- 启动健康检查验证公开 URL、API、数据库和 Worker。
- Open Graph、manifest、canonical 与语言 alternate 在预览环境自动测试。

## 7. 必须先解决的产品规则冲突

原 V1 文档明确规定：

- 活动费由组织者在 App 外收取。
- Spott 不代收、不结算、不担保、不抽成。
- 禁止出现平台收款、代退和按活动费比例抽成。
- V1 不包含用户私信和无限制群聊。

新目标要求覆盖 Luma 与 Meetup 的全部核心能力，因此必然涉及：

- 平台订单、票种、支付、税费、退款、拒付和结算。
- 会员订阅或会费。
- 私信、活动聊天和内容治理。

处理原则：

1. 新目标不应由开发者通过代码静默覆盖旧产品规则。
2. 在进入商业交易和私信阶段前，产品负责人必须批准新的资金边界、责任边界、隐私规则、内容治理规则和运营流程。
3. 数据模型可以先采用提供方无关的抽象，但生产收款前必须完成日本适用的支付、税务、消费者保护和平台责任审查。
4. 审批前，现有 App 外收款流程继续保留，不伪装成平台支付。

## 8. “比竞品更好”的产品定义

功能数量本身不能证明领先。Spott 的领先必须由以下结果验证。

### 8.1 参与者体验

- 访客无需安装即可在 Web 看清活动核心信息。
- 首次报名在资料齐全时 90 秒内完成。
- 报名、候补、支付、提醒、签到、反馈状态在 App 与 Web 实时一致。
- 活动语言、主办方可沟通语言和自动翻译状态清楚可见。
- 精确地址只在符合隐私规则的状态下展示。
- 取消、退款、活动变化和安全风险始终有明确后续动作。
- 从活动结束到加入群组、聊天、相册和下次活动形成自然闭环。

### 8.2 组织者体验

- AI 辅助下，合格活动可在 5 分钟内完成首个可审核草稿。
- 可复制或创建周期活动，并只修改单场例外。
- 一个工作台完成票种、报名问题、审批、名单、通知、现场签到、退款和复盘。
- 可以邀请共同主办、编辑、营销和签到工作人员，并使用最小权限。
- 关键指标给出解释和建议，不只展示数字。
- 所有高风险操作可撤销、可审计或需要二次确认。

### 8.3 日本与跨语言优势

- 地址、车站、都道府县、日元、JST、姓名和假日语义符合日本用户习惯。
- 中文、日语和英语不是三个独立产品；同一活动可清楚展示原文和翻译。
- 主办方可以指定主要语言、可支持语言和现场翻译条件。
- 安全说明、集合方式、取消规则和紧急信息优先保证翻译质量。

### 8.4 信任安全优势

- 组织者信誉由真实完成活动、到场、取消、纠纷和响应行为形成，不与消费金额绑定。
- 对高风险、未成年人、夜间、户外、水上、饮酒活动采用分层规则。
- 用户可在不公开精确地址和隐私信息的情况下判断是否值得参加。
- 举报、证据、限制、申诉和恢复过程对用户透明且可追踪。

## 9. 高级、精致、克制的跨端设计方向

### 9.1 品牌表达

推荐设计语义：Quiet Confidence，安静、自信、可信、有温度。

- 不是夜店海报式的高饱和活动平台。
- 不是企业 SaaS 式的灰色表格集合。
- 视觉应接近日本城市生活方式编辑产品：温暖中性色、清楚网格、高质量活动图片、克制强调色、明确层级。
- 品牌个性来自内容策展、排版、空间、交互动效和可信信息，而不是堆叠渐变、玻璃、阴影或装饰。

现有 design tokens 中的 warm canvas、twilight、coral、mint 可继续使用，但必须补齐语义层、交互状态、暗色对比度和平台映射。

### 9.2 设计原则

1. 每屏只有一个明显主操作。
2. 活动图片服务于判断，不掩盖时间、地点、费用和名额。
3. 复杂设置渐进展开，默认值可解释并可恢复。
4. 重要状态使用文本、图标和颜色共同表达。
5. iOS 尊重原生导航、手势、Sheet、Dynamic Type 和触觉。
6. Web 尊重 URL、浏览器返回、键盘、焦点、响应式和可分享性。
7. 动效解释空间和状态变化，不制造等待。
8. 三种语言从组件设计开始验证，不在发布前补字串。

### 9.3 组件与版式验收

| 维度 | 标准 |
| --- | --- |
| 触控目标 | iOS 与 Web 直接交互区域均至少 44×44 |
| 栅格 | 4/8pt 基础；页面间距、卡片内距和内容宽度使用令牌 |
| 字体 | 支持中、日、英对应字体；数字、日期和金额层级一致 |
| 对比度 | 正文和控件满足 WCAG 2.2 AA；焦点和错误状态不只靠颜色 |
| 圆角 | control、card、cover、panel 使用固定语义，不随意新增 |
| 阴影 | 只用于层级和浮层；普通卡片优先边界与背景差 |
| 图片 | 明确裁切、焦点、加载、失败、版权和无障碍替代文本 |
| 动效 | fast 120ms、standard 220ms、narrative 360ms 为基础，并支持减少动态效果 |
| 状态 | loading、empty、error、offline、permission、success 均有统一组件 |
| 表单 | 标签常驻、错误贴近字段、输入不因失败丢失、长流程可恢复 |

### 9.4 核心页面设计重点

- 发现页：降低 hero 高度，让真实活动在移动首屏出现；地图与列表共享过滤状态。
- 活动详情：首屏回答是什么、何时、哪里、多少钱、谁组织、还剩多少名额、使用什么语言。
- 报名：一次只处理一类决策；费用与取消规则在提交前可复核。
- 创建：保留六步结构，增加进度、自动保存、冲突提示、AI 辅助和完成度评分。
- 组织者工作台：以“下一步行动”组织，而不是以数据库状态堆表。
- 群组：区分公告、讨论、聊天和相册，防止所有内容挤进单一时间线。
- 现场模式：大字号、高对比、弱网可用、单手操作、快速撤销。

## 10. 目标架构与跨端对等规则

### 10.1 保持模块化单体，逐步扩展

现阶段继续使用 NestJS 模块化单体、PostgreSQL、OpenAPI、Worker 和 Outbox。没有证据支持拆微服务。应新增清楚的领域边界：

- EventSeries 与 EventOccurrence
- EventTeamMember 与角色权限
- TicketType、Order、Payment、Refund、Coupon、TaxLine
- Calendar、MembershipPlan、Subscription
- InvitationCampaign、AudienceSegment、Delivery
- Conversation、Message、ModerationState
- Album、Photo、MediaPermission
- ExportJob
- PublicApiClient、WebhookEndpoint、WebhookDelivery

所有新增表使用追加迁移、外键、状态约束、审计字段和幂等键。

### 10.2 跨端能力对等

对等不要求两个平台长得一样，但要求用户能力和事实一致：

- 每个产品能力在 OpenAPI、API、数据库、Web、iOS、运营端、通知、埋点和测试中有明确适用项。
- App 与 Web 不允许各自解释状态机。
- 服务器负责权限、容量、时间、价格、筛选和最终状态。
- 客户端可以优化交互，但不能制造假成功、假坐标、假库存或假支付。
- 某平台暂未开放的能力必须由服务端 capability flag 明示，并展示真实说明。
- 任何功能若只有一个客户端完成，仍标记为“部分”，不能计入对等完成。

### 10.3 文件拆分方向

随相关功能开发逐步执行：

- GroupsViews.swift 按目录、详情、成员、公告、转让和解散拆分。
- ProfileViews.swift 按公开资料、本人设置、组织者、积分和安全拆分。
- APIModels.swift 按领域分组，生成模型与手写展示模型分离。
- SpottAPIClient.swift 拆为传输、认证刷新、媒体上传和领域服务。
- EventComposer.tsx 拆为状态机、schema、步骤组件、上传和提交适配器。
- globals.css 拆为 tokens、base、layout、components、utilities 和 route-specific 样式。

- 受众分组、邀请、定时公告、邮件模板与退订。
- 现场快速签到、工作人员设备、弱网恢复。
- 更完整的活动洞察、来源和到场漏斗。

退出标准：

- 周期活动可修改全部后续或仅改单场。
- 权限矩阵在 API、Web、iOS 和 Ops 一致。
- 邀请和批量操作具备频率、隐私、退订和审计保护。
- 组织者可在一套工作台完成活动前、中、后流程。

退出标准：

- 支付提供方回调重复、乱序和延迟不会重复出票或退款。
- 订单、票、支付、退款和结算可审计且可对账。
- App Store 数字商品与线下活动票务边界清楚。
- 法务、税务、消费者保护和客服流程通过负责人验收。

### Phase 3：Meetup 级社区关系

目标：让一次参加沉淀为长期可信关系，而不是只完成报名。

范围：

- 群组讨论主题、回复、提及和通知。
- 活动临时聊天与长期群聊，支持关闭、静音和主持。
- 用户私信请求、陌生人权限、举报、屏蔽和反骚扰。
- 活动相册、投稿、审核、版权与删除。
- 群组会员、会费、成员标签和入群问题。
- 多群组组织、品牌主页、活动模板和跨群组分析。

退出标准：

- 私信与群聊具备默认隐私、频率限制、举报和证据链。
- 内容删除、拉黑和封禁在所有端即时生效。
- 群组负责人能管理成员、内容、活动和角色。
- 用户可清楚控制谁能联系自己。

### Phase 4：Spott 领先能力

目标：不是复制竞品，而是在日本和跨语言场景建立 L5 优势。

范围：

- AI 活动创建助手：根据目标生成结构化草稿、风险提示和三语文案，所有内容可编辑。
- 跨语言活动页：原文、翻译、现场语言和关键规则的可信展示。
- 个性化发现：结合兴趣、距离、时间、语言、履约与多样性，提供可解释推荐。
- 防爽约与候补优化：基于用户明确同意的数据给出提醒策略，不进行歧视性自动拒绝。
- 主办方 Copilot：总结问题、预测运营风险、生成分群通知和复盘建议。
- 安全同行、到场确认和活动后关怀等可选安全工具。

退出标准：

- AI 输出始终标记、可编辑、可追溯，不自动发布高风险内容。
- 推荐和风控支持解释、申诉、人工覆盖和偏差监测。
- 领先能力提升报名转化、到场率、复购或组织效率，并通过 A/B 或队列实验验证。

## 12. 质量与体验硬门槛

### 12.1 性能

- Web Core Web Vitals p75：LCP 小于 2.5s、INP 小于 200ms、CLS 小于 0.1。
- Web 首屏不下载非当前路径所需的大型脚本和图片。
- iOS 冷启动目标小于 1.5s；关键列表正常设备维持 60fps。
- 图片按设备尺寸提供衍生图，禁止列表加载原始大图。
- API 关键读请求 p95 小于 400ms，写请求 p95 小于 800ms，不含外部支付等待。

### 12.2 稳定性

- 报名、候补、签到、积分、订单、退款全部幂等。
- 关键用户流程成功率至少 99.5%。
- 移动端 crash-free sessions 至少 99.8%。

### 12.3 安全与隐私

## 14. KPI 与验收仪表盘

### 北极星指标

每周“真实完成活动并产生至少一次后续关系行为”的用户数。后续关系行为包括加入群组、关注主办方、参与讨论、查看相册或报名下一场活动。

### 参与者漏斗

- 发现曝光 → 详情查看
- 详情查看 → 开始报名
- 开始报名 → 报名完成
- 报名完成 → 实际到场
- 到场 → 反馈
- 到场 → 30 日内再次参加

### 组织者漏斗

- 开始创建 → 有效草稿
- 有效草稿 → 发布
- 发布 → 首次报名
- 首次报名 → 活动完成
- 活动完成 → 30 日内再次创建

### 质量护栏

- 取消率、爽约率、投诉率、退款争议率
- P0/P1 事故数
- 三语缺失率
- 核心流程错误率和崩溃率
- 活动供需地区集中度
- 推荐和风控不同语言、地区、年龄段的偏差指标
