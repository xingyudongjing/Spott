## 1. 执行结论

Spott 已经明显越过原型阶段，核心活动平台的本地代码骨架和多条主流程已经成型；但它还不是“开发完成、可以上线”的产品。最准确的结论是：**功能代码覆盖较高，Definition of Done 闭环中等，生产上线准备偏低**。

| 口径 | 中心估计 | 合理区间 | 结论 |
| --- | ---: | ---: | --- |
| 产品文档 C–T 功能覆盖 | **70%** | 68%–72% | 大部分 V1 主流程有真实代码，但跨端与完整体验仍有断口 |
| 开发计划 DoD 闭环 | **58%** | 56%–62% | 代码与测试资产很多，但当前门禁红、跨端 E2E 和生产门未闭环 |
| 生产上线准备 | **26%** | 22%–30% | 真实短信、推送、StoreKit、云基础设施、SLO、合规与发布证据缺失 |
| 计划文档复选框回填 | **22%** | 固定计数 | 116 已勾 / 412 未勾；反映计划维护状态，不等于代码完成度 |

按附件评分表的 18 个领域、120 总权重复算，三项加权原始值分别为 **69.84%**、**58.27%** 和 **26.34%**；上表为便于管理决策的四舍五入中心值。

### 1.1 一句话判断

**如果只问“代码里有没有主要功能”，大约完成七成；如果问“是否按开发文档的 Definition of Done 完整交付”，大约完成六成；如果问“能否安全稳定地正式上线”，目前约四分之一。**

### 1.2 为什么不能沿用总纲里的“工程门禁真绿”

当前总纲记录的是旧快照，并且本次新鲜验证已经推翻了其中的部分结论：

- 总纲写 `pnpm check` 和 `test:quality:ci` 均通过；本次无缓存验证两者都未形成全绿结论。

- 因此，总纲可以继续作为路线图，但不能作为当前完成度的唯一验收证据。

### 2.3 四轴评分

每个产品领域按四条证据轴打 0–100 分：

1. **实现覆盖**：是否有真实、连通的主线代码，而非模型、页面壳或演示数据。
2. **自动验证**：当前 HEAD 是否有单测、集成测试、契约、构建与门禁证据。
3. **跨端验收**：iOS、Web、API/Ops 是否形成同一用户旅程，并有当前运行证据。
4. **生产就绪**：真实外部服务、基础设施、SLO、监控、恢复、合规和发布门是否完成。

产品功能覆盖公式：

`70% × 实现覆盖 + 30% × 跨端验收`

开发计划闭环公式：

`25% × 实现覆盖 + 30% × 自动验证 + 25% × 跨端验收 + 20% × 生产就绪`

开发文档的 DoD 明确要求跨端验证、可观测、无障碍、本地化、回滚、Runbook 与验收证据，因此不能把“代码存在”直接计为 100%。

### 2.4 领域权重

产品文档 C–T 共 18 个领域，权重合计 120。报名履约、活动发布、积分正确性、安全、非功能和 P0 发布验收权重更高；成就、分享和配置等增强能力权重较低。完整明细见 `docs/quality/local-completion-audit-20260718.csv`。

## 6. 尚未完成的内容

### 6.1 P0：当前必须先关闭的发布阻断

1. **质量门不是真绿**  
   API 有 1 个时间依赖失败；Web unit 有 28 个失败；质量 manifest 有 1 个 worktree 路径失败；强制无缓存全仓门因 Worker 并发媒体测试超时失败。

2. **Web 会话安全切换未完成**  
   `apps/web/app/lib/client-api.ts` 仍把 access token 和 refresh token 一起序列化到 `localStorage`。后端 BFF/HttpOnly 基础代码很多，但客户端没有完成切换。

3. **Ops MFA 未完成**  
   主分支后台依赖邮箱 OTP，并把 `mfa_enrolled_at` 当成 MFA 已验证；真实 TOTP 在 `fix/ops-mfa` 分支，但因注册引导可被被盗邮箱会话自助完成而未合并。

4. **iOS 可恢复启动未完成**  
   `Spott/Core/Persistence/PersistenceStore.swift` 初始化 SwiftData 失败后仍会 `fatalError`，不符合开发文档的可恢复错误要求。

5. **十条 P0 场景没有当前 HEAD 的整套验收**  
   代码组件分散存在，但没有同一数据库、同一构建、Web 与 iOS 连续跑通的证据包。

6. **生产外部服务未接**  
   `.env.example` 中 OTP、Push、Email 仍以 console 为默认；APNs 凭证为空；StoreKit 商品与 Sandbox、生产对象存储、OAuth、域名和 Associated Domains 未验证。

### 6.2 P1：代码闭环与产品质量缺口

- Ticketing shell 有迁移、API controller/service 和测试，但没有进入 OpenAPI，也没有 Web/iOS 票种、优惠码和嘉宾体验，不能按跨层完成计分。
- 新版群组 discussion 后端端点没有进入 OpenAPI；Web/iOS 当前主要还是公告与受控评论路径。
- 分类成就缺稳定 category slug；可信局头成就缺真实认证流程。
- Web 同步只可靠持久化游标，尚未形成完整的浏览器离线业务缓存和受控恢复能力。
- iOS 三语资源键整齐，但仍检测到约 367 行疑似直接中文 UI 字面量；Web/Ops TSX 检测到约 1,090 行含中文文本。这是启发式计数，不代表每一行都是缺陷，但说明三语闭环不能只看键数量。
- Web 构建存在大于 500 kB 的 chunk 警告；OpenAPI 存在 `/media/upload-attempts/{attemptId}` 与 `/media/{id}/complete` 歧义警告。
- 没有当前真机 / 模拟器的 VoiceOver、Dynamic Type、深色模式、弱网、帧率、冷启动和内存证据。
- 视觉令牌和部分改版已落地，但没有产品负责人签字的截图矩阵、Figma 规范或最终视觉验收。
- Analytics 目前是代码和局部指标，不是已运行的数据平台；没有真实业务看板、告警、留存与数据质量监控。

### 6.3 P2 / 战略能力缺口

这些不一定都属于当前 V1 必做，但它们直接决定“超越 Luma + Meetup”是否成立：

- 周期活动 / series / RRULE 与单场例外。
- 线上与混合活动、受控会议链接和 Zoom 类集成。
- Calendar / 组织中枢、订阅和多主办方协作。
- 活动主题 / 外观自定义、无图品牌封面和高级视觉编辑能力。
- 相册、联系人库、批量邀请、Newsletter 与营销编排。
- 会员、订阅、组织者网络、发票和平台支付；其中多项需要先改变“不代收”边界并做日本法律评估。
- 公开 API、Webhook、自定义域名和嵌入式 widget。
- Android 原生客户端；当前范围实际是 iOS + Web。

因此，**“已证明超越 Luma + Meetup”目前不能成立**。代码已经形成若干差异化优势，但没有用户研究、留存、性能、可访问性和生产数据证明整体领先。

### 7.1 失败性质分析

- API 推荐流失败是**时间依赖测试**：用 `new Date() + 3 小时` 构造“今天”的活动，在东京深夜会跨到次日，`today` 模块为空。
- Web 28 个失败主要是**固定日期 fixture 过期**：活动固定在 2026-07-18，运行到活动结束后，详情和报名流正确地进入“报名已结束 / 暂不可用”，测试却仍期待可报名。
- Worker 媒体测试在全仓并发门内超时，但单独运行 35/35 通过，说明问题更接近门禁并发稳定性和 5 秒预算，而非媒体逻辑必然错误。
- Quality manifest 失败来自仓库中的 `.claude/worktrees/...` 被发现为额外集成测试来源，说明当前清单扫描与本地嵌套 worktree 拓扑不兼容。

这些失败里有多项属于测试基础设施问题，不代表对应产品功能一定坏了；但在开发文档的 DoD 口径下，**门禁不能稳定通过本身就是未完成**。

## 12. 建议的下一阶段开发顺序

完成标准：干净 checkout 上无缓存 `lint + typecheck + unit + build + quality manifest` 稳定重复通过。

## 16. 最终判断

Spott 当前最有价值的部分不是页面数量，而是已经形成的**活动履约、信任安全、积分账本、跨端同步和多语言产品骨架**。这些基础值得继续演进，不建议重写。

当前最大问题也不是“还有多少功能没写”，而是：

- 文档快照与真实门禁已经漂移；
- 一些关键安全与恢复能力仍停在基础层；
- 跨端场景没有用当前 HEAD 完整验收；
- 生产外部服务、运维与合规几乎还没进入交付阶段。

因此，当前应把项目对外状态定义为：

> **V1 核心功能代码约完成 70%，工程 DoD 约闭环 58%，尚未达到发布候选版本；完成当前门禁、安全恢复、十条跨端 P0 与生产前置后，才可以进入“可上线”判断。**
