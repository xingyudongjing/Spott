- API 的 accepted 权威已改为 disposition 与 outcome 在 attempt/challenge/device/binding/generation/session 上完全相等的精确连接；删除 outcome 后 authority/bootstrap/refresh/logout/upgrade 均 fail closed。
- Worker 仅以 disposition 为完成态事实源；pending 到期做精确 revoke/discard，accepted 只在 session 不活跃后清理，并有 `SKIP LOCKED`、竞争与公平性 PG 覆盖。
- 共享尝试账本已经完成：`localStorage` 单一 metadata-only 记录，最多 1 个 unresolved attempt，显式 readable/blocked inventory，不静默删除、不过期淘汰、不存 secret，阶段为 `prepared → reconciled → device_committed → accepting`，全部 exact CAS。
- 共享账本直接测试在 Node 24 下 27/27 通过，定向 ESLint 和 `git diff --check` 通过。

### 4.2 独立安全复审发现的必须关闭项

1. **Critical：跨标签页登出后会话复活。** 旧 attempt authority 在单标签页 `sessionStorage`；B 标签页可完成登出并清 tombstone，A 标签页随后恢复 `accepting` 并重新认证。
2. 浏览器可读 completion bearer 曾进入 Web Storage；XSS 或复制值可离开浏览器重放。
3. 旧扫描会静默删除过期/损坏记录并在容量满时淘汰 unresolved authority。
4. 120 秒 prepare 窗与 accepted 响应恢复/30 天保留窗错位；边界响应可能成为不可恢复的活跃会话。
5. BFF、浏览器和 API 时钟/配置缺少包含关系和有限 clock-skew 合同。
6. accepted gen0 material 在 refresh gen1+ 或 binding upgrade 后不能安全回放；长期能力必须有 exact server-side revoke，而不是重发陈旧 gen0 Cookie。
7. pending→accepted 决策必须重新锁定并验证 user 未删除/active/无 `loginBlocked`、device 存在且未 blocked。
8. 旧 token 的 KID 允许 `.`，但点分序列化无法解析 dotted KID。

### 4.3 已锁定的新协议

- 浏览器 Web Storage 不保存任何 completion bearer、binding proof、access token 或 refresh token。
- `localStorage` 只保存一个全浏览器共享、非敏感、exact-CAS 的 attempt 元数据记录。
- 真正恢复能力是单一 HttpOnly Cookie：`__Host-spott_login_intent`，必须 `Path=/; HttpOnly; Secure; SameSite=Strict`。
- 单例 Cookie **绝不能覆盖**任何有效 prepare/reconcile Cookie。相同 prepare 可幂等返回 ready；不同 challenge/device 或 reconcile+新登录必须返回 in-progress/conflict。
- 第一次 `/api/session/complete` 只返回 `{state:'completion_ready', attemptId, expiresAt}`；不向 JS 返回 token。
- 第二次 `/api/session/complete` body 使用 `attemptId`，BFF 从 HttpOnly Cookie 取权威；pending 返回 `reconcileExpiresAt` 并把 Cookie 旋转到长期 reconcile phase。
- Set-Cookie 已应用但 JS 丢失 body 时，同一 `attemptId` 可从 reconcile Cookie 幂等恢复 exact pending。
- accept/discard body 只含 `{attemptId}`，BFF 以 HttpOnly Cookie 恢复 binding proof 并签名调用 API。
- 登出意图一旦存在，普通 completion/disposition 不得发布新会话。
- BFF logout 必须先处理 HttpOnly completion capability：pending 做 exact discard；accepted 调用 API exact revoke；不能把 accepted material 或陈旧 gen0 Cookie 发到浏览器后再登出。
- API 需要新的 attempt-authorized server-side revoke：用不可变 disposition sessionId 原子撤销当前 session、当前/旋转后的 refresh history 和 binding；重复调用幂等，绝不返回 session material。
