# Task 22: Recoverable, Idempotent Media Upload Execution Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task by task, and use `superpowers:test-driven-development` for every behavior change. Do not mark the slice complete until the response-loss and cross-owner PostgreSQL tests pass.

**Goal:** Make media intent creation, object upload, completion, abandonment, and later attachment recoverable without duplicate assets, cross-account leakage, mutable hashes, or false submission success on Web and native iOS.

**Architecture:** A caller owns one UUID attempt per stable file instance and stage, and the upload `attemptId` is exactly the canonical `Idempotency-Key` the caller already knows before the first request. The API durably stores the decoded expected SHA-256 separately from the request fingerprint and binds the attempt to the current owner + immutable normalized payload while retaining the creating owner only as audit history. Clients upload through a Spott-owned bounded streaming gateway, never a provider presigned URL: before receiving bytes, the gateway atomically claims one durable upload lease for the asset/capability generation. It then receives exact bytes into a crash-recoverable private temporary file under a hard total deadline, verifies size and SHA-256, and only then performs one private provider PUT to that lease's unique staging key under a separately enforced outbound deadline. The asset atomically points at the one winning immutable staging object; there is no staging-to-canonical copy. Completion and every attachment/report mutation use separate caller-retained UUIDs plus lifetime durable receipts. Object-store writes are fenced with lease-specific keys and durable cleanup work; abandonment preserves a tombstone until every gateway and provider write fence has closed and final absence is verified.

**Stack:** PostgreSQL 18, NestJS/Zod, OpenAPI/Redocly, Vitest, React/Vitest/Playwright, native Swift/CryptoKit/XCTest/XCUITest.

## Global Constraints

- `attemptId == Idempotency-Key` after canonical lowercase UUID parsing. An upload attempt UUID belongs to exactly one authenticated owner and one normalized intent payload; the server never invents a second recovery identifier.
- The normalized intent includes `purpose`, NFC filename, MIME, byte size, focal point, and lowercase 64-character SHA-256. Changing any field requires a new intent attempt.
- The decoded expected SHA-256 is stored in `expected_content_hash bytea` with an exact 32-byte constraint. `intent_request_hash` remains a one-way idempotency fingerprint and must never be used as a substitute for the recoverable expected digest. Verified `content_hash` remains null until completion succeeds.
- File identity is not inferred from filename, size, timestamp, or object key. Web keeps identity by actual `File` instance; iOS keeps an explicit selection instance UUID.
- Intent replay or lookup returns the same asset ID. Before commit, each capability generation owns a different private staging key; terminal failure rotates generation and therefore rotates only that staging key. The asset's authoritative object key/version/checksum remain null before the winning CAS, are set exactly once to the committed generation's key/version/checksum, and are immutable afterward. Recovery of a committed upload returns its receipt and never issues another capability.
- Credential-bearing intent/recovery responses always use `Cache-Control: no-store`, `Pragma: no-cache`, and `Referrer-Policy: no-referrer`; state/receipt-only responses remain non-cacheable but contain no capability fields. Gateway URLs/capabilities and required headers never enter analytics, service-worker caches, browser persistence, logs/diagnostics, the DOM, accessibility tree, traces/HAR/network snapshots, or application-generated screenshots. User/OS screenshots are outside application control and are not claimed as preventable.
- The durable idempotency row stores only the normalized request fingerprint plus a safe replay stub (`attemptId`, `assetId`, authoritative state, and resource reference) committed in the same transaction as the asset. It never stores a gateway URL/capability, signed query, authorization header, object key, or other reusable upload credential in `response_body`. Only a pending/uploadable asset with no active lease, or recovery after terminal failure plus confirmed provider abort/object absence and generation rotation, materializes a fresh capability after commit. Active, committed, uploaded, processing, ready, rejected, abandoned, and deleted states return only authoritative state/receipt and no capability. A crash after commit therefore replays the stub instead of remaining `REQUEST_IN_PROGRESS` until expiry.
- The upload credential is an opaque, expiring gateway capability bound to the exact attempt, asset, current owner ID, method, MIME type, byte length, expected SHA-256/checksum header, generation, and expiry. Provider object keys and credentials never reach clients. It cannot authorize another attempt or an unbounded body.
- Completion has its own attempt UUID. The first accepted content hash is immutable; a different hash is a terminal conflict even with another completion key.
- Only the owner can recover, renew, complete, or abandon an asset. Cross-owner probes return the same not-found envelope and reveal no existence, state, object key, filename, or hash.
- Completion enqueues `media.processing_requested` once for the asset's lifetime, including after the event has been published and generic idempotency records have expired. A response replay never duplicates historical or active outbox rows.
- Upload completion does not mean media moderation is approved, attachment succeeded, or a report/event/profile/group mutation succeeded.
- Replacing or removing a local file abandons only an unreferenced asset owned by the current account. Account switching never reuses or mutates the previous owner's attempt.
- Cleanup never deletes an asset referenced by event media, a profile, a group, ordered report evidence, or a poster job. Cleanup is auditable and object deletion is retryable.
- Report evidence locks assets in stable ID order and requires reporter ownership, `report_evidence` purpose, verified completion, and non-deleted state. Missing and cross-owner assets share the same safe 404. Evidence rows retain user-visible order and store the verified media digest, never a digest of the asset UUID.
- Complete, attach, report submission, abandon, cleanup, gateway commit, and worker finalization share one asset-row lock/CAS protocol. Every domain attachment/report locks the asset before inserting a reference; abandon/cleanup lock the same row before rechecking every reference. Event/profile/group attachment, arrangement, and report-submission keys are stable caller-retained UUIDs with lifetime durable receipts; they replay the exact committed response after generic idempotency expiry without duplicate version increments, reports/cases, outbox rows, or old-asset deletion.
- A worker claim owns an exact monotonically increasing generation plus random lease ID. Before any external derivative PUT it persists cleanup intent and writes only to a lease-scoped private staging key. Promotion/finalization, retry, reject, and failure all compare-and-set the same lease + state + row version and check `rowCount`; a stale worker can neither publish `media.ready` nor leave an untracked object.
- Each issued gateway capability monotonically extends `latest_authorization_expires_at`. Deadline enforcement uses two explicit clock domains. At the first route-handler instruction, start a process-monotonic elapsed timer for the configured total inbound duration; this timer covers DB-lock wait, header-complete/time-to-first-byte, streaming, and verified EOF. Under the asset lock, subtract monotonic elapsed time; if no budget remains, fail before any reservation/file. Otherwise use PostgreSQL `clock_timestamp()` as the cross-process authority and persist `inbound_deadline_at = db_clock_timestamp + remaining_budget` in the durable lease before reserving bytes, opening a file, or waiting for a body byte. At most one lease for a generation may exist. A concurrent request returns authoritative in-progress/committed state and never opens another file/provider PUT. After verified EOF and before provider PUT, start a new process-monotonic outbound timer and persist `provider_deadline_at = db_clock_timestamp + configured outbound duration` (bounded by any persisted lease hard cap); both local timer and adapter abort enforce it, and abort confirmation is persisted. These DB `timestamptz` fence instants are comparable across restart/instances and are never confused with monotonic epochs. Abandon prohibits renewal under the asset lock. `cleanup_not_before` is SQL `GREATEST` across `latest_authorization_expires_at + validated clock skew`, every persisted `inbound_deadline_at`, every persisted `provider_deadline_at`/abort-confirmed instant, and every DB worker lease expiry. Missing abort confirmation keeps cleanup open. The gateway cannot begin a provider PUT until the complete body passed deadline/size/hash validation; provider writes use the lease's unique staging key whose cleanup row exists first. Success CASes asset row version + capability generation + lease ID to one immutable authoritative object key/version/checksum and a committed upload receipt; the asset points directly to that winning staging object, so no copy/promotion race exists. Failed/stale leases may delete only their own versions/markers. Cleanup performs HEAD → delete all versions/markers → HEAD, retains a tombstone, and rechecks after the exact last enforced write fence has closed. Unsupported version enumeration or abort confirmation fails closed.
- Object-store metadata/read/write/delete operations sit behind injected, fail-closed private adapters. Production never falls back to a syntactically plausible local URL when provider configuration, checksum enforcement, version policy, metadata verification, hard request deadlines, abort semantics, or deletion authority is absent. The bucket has no anonymous write and no client-facing CORS surface. ETag is never accepted as SHA-256 evidence.
- Ownership has exactly two meanings: `current_owner_id` is the only authorization, recovery, uniqueness, capability, and lifetime-receipt scope; immutable `created_owner_id` is audit-only and never authorizes access. Account merge uses one strategy: a preflight transaction locks source/target identities, media assets, media lifetime receipts, gateway leases, and live media idempotency rows in canonical UUID order. If source ownership transfer would collide on upload attempt or `(operation_type, idempotency_key)`, it persists `blocked_media_collision` on the merge job plus an audit event and performs no owner/media/idempotency mutation. With no collision, asset and lifetime receipt `current_owner_id` values migrate atomically to the target, every affected capability generation increments so all source capabilities become invalid, and source generic media idempotency rows retire only after durable target-scoped replay is proven. `created_owner_id` never changes.

## Endpoint contract

1. `POST /media/upload-intents`
   - Requires `Idempotency-Key` UUID.
   - Requires the existing media fields plus `contentSha256`.
   - A first `201`, or an eligible `200` replay while the asset is pending/uploadable with no active lease, returns `attemptId` equal to the canonical request key, `assetId`, authoritative `state`, same-origin Spott gateway URL, opaque capability/required headers, `expiresAt`, `maxBytes`, and hard total request deadline. Active/committed/uploaded/processing/ready/rejected/abandoned/deleted replay returns authoritative state/receipt only and the capability fields are absent from the response schema.
   - Same owner/key/payload returns the same asset. Same owner/key/different payload returns `IDEMPOTENCY_KEY_REUSED`.
2. `GET /media/upload-attempts/{attemptId}`
   - Owner-scoped recovery after a lost intent response.
   - Returns the same asset and current state plus authoritative gateway-lease state. A committed upload receipt is returned without starting another provider PUT; an active lease returns `in_progress`. Only after a failed lease has terminal abort/absence confirmation may recovery rotate capability generation and issue a fresh capability from durable `expected_content_hash`, preserving the asset and atomically extending `latestAuthorizationExpiresAt`.
   - Returns no gateway capability after uploaded/processing/ready/rejected/deleted.
   - Every credential-bearing response is non-cacheable and never exposes a provider object key or credential.
3. `PUT /media/upload-attempts/{attemptId}/content`
   - Requires the exact opaque gateway capability plus bound MIME, byte length, and SHA-256 header; it does not authenticate with browser cookies.
   - Claims the single durable lease for the capability generation before accepting bytes. A duplicate concurrent call returns the authoritative in-progress/committed outcome and never opens another temp file or provider request.
   - Enforces one absolute inbound deadline from handler entry/header completion through first byte and verified EOF, writes only to a no-follow, exclusive mode-0600 file under a dedicated mode-0700 gateway directory with a durable temp manifest, and does not open a provider request until size/hash pass. A client that sends no first byte by the same deadline is terminated and reconciled.
   - Performs the private provider PUT to that lease's unique staging key under a separate hard abort deadline after durable cleanup intent exists. Success CASes the asset to that exact object version and commits the upload receipt; timeout/error removes only its own temp/staging artifacts and leaves retryable tracked cleanup work.
4. `POST /media/{id}/complete`
   - Requires a distinct `Idempotency-Key` and `X-Content-SHA256` matching the intent.
   - First completion transitions `pending_upload -> uploaded` and emits exactly one outbox event.
   - Same hash replay returns the authoritative state; different hash returns `MEDIA_HASH_MISMATCH` without mutation.
5. `DELETE /media/{id}`
   - Requires its own `Idempotency-Key`.
   - Idempotently marks only an owned, unreferenced asset as deleted/abandoned, permanently disables renewal, and queues final object cleanup at the greatest concrete instant across latest issued capability plus skew, every gateway inbound deadline, every provider deadline/abort confirmation, and every worker lease expiry.
   - Referenced assets return a safe conflict; callers must use the owning domain's replace/detach operation.
   - An active processing lease returns a retryable conflict. After lease expiry, cleanup may take the row lock, mark deletion authoritatively, and a stale worker must fail its final compare-and-set while its staged/object writes remain covered by durable cleanup work.
6. `POST /media/{id}/attach/event/{eventId}`, `POST /media/{id}/attach/profile`, and `POST /media/{id}/attach/group/{groupId}`
   - Each requires its own stable `Idempotency-Key`; the upload intent/completion key is never reused.
   - Claim, asset/domain locks, mutation, sync/outbox writes, previous-asset handling, and replay response commit in one transaction.
   - Event gallery reorder uses one idempotent batch arrangement mutation. Reordering existing assets rotates only the arrangement key and never rotates upload/completion attempts.
7. Report submission
   - Keeps its existing separate stable submission key. It locks all evidence assets in stable ID order, validates owner/purpose/completion/deletion under those locks, stores real verified hashes and order, then atomically commits the report, safety case, outbox, and a lifetime `report_submission` mutation receipt once.
